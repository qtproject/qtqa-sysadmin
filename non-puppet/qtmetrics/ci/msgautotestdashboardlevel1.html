AUTOTEST DASHBOARD DESCRIPTION:

Level 1: Show list of failing Autotests

GENERAL
Purpose Improve Continuous Integration (CI) stability, throughput and coverage by fixing Autotests e.g. to make insignificant Autotests significant
Question Which Autotests are failing most often within a same project and/or configuration build
 
METRIC
Filters Project, Configuration, Autotest, Timescale
Indicator List of failing Autotests by failure category
Tip: You can sort the list by clicking the desired column title.
Calculation In latest builds: Counting the number of failures in each Configuration (and Project) in Project specific latest build for each Autotest (by failure category)
In all builds: Counting the total number of failures in each Configuration (and Project) in all builds for each Autotest. Calculated either based on all data in the database or according to the Timescale filter i.e. based on the builds done after the selected date. Note: This is calculated only when a Project or Configuration, or the Timescale, is filtered.
Interpretation Higher count means the Autotest is failing in higher number of Configurations, and should therefore be analyzed
Target value Zero i.e. not listed here
 
DATA
Data items Project: e.g. QtBase_stable_Integration
Build number: several per Project (numeric 1 to n; number is Project specific)
Configuration: several per Project Build (e.g. linux-g++-32_Ubuntu_10.04_x86)
Configuration significance: number of blocked/insignificant Configurations (if Configuration is tagged as "insignificant" the build will not fail even though some significant Autotests would fail)
Autotest name: several per Configuration (e.g. tst_qftp)
Autotest result: name in the list indicates a failure (i.e. the passed ones are not listed)
Autotest significance: number of failed Autotests with tagging Significant (failure leads to build failure, unless the Configuration set insignificant) / Insignificant (failure does not lead to build failure)
Data source Data in SQL database from log files in http://testresults.qt-project.org/ci/
Data update cycle New builds are updated to database in real time
